## Context

"去K书"自习室预约系统已完成用户认证模块（注册、登录、短信验证）。当前首页（`pages/index/index.vue`）仅有登录/登出功能，无任何业务数据展示。V1 版本需要首页作为核心入口，参考高保真原型 `prototype/home.html` 实现，引导用户浏览自习室并进入预约流程。

技术栈：uni-app (Vue3) + uView 2.0 前端，FastAPI + SQLAlchemy 后端，PostgreSQL 18 数据库。

## Goals / Non-Goals

**Goals:**
- 首页顶部导航栏：定位（城市选择）+ 搜索框 + 通知铃铛
- 首页展示轮播图运营位：带标题/副标题/CTA 按钮，支持点击跳转，3.5 秒自动轮播
- 快捷入口网格：钱包充值、卡券套餐、美团兑换、个人码（V1 跳转占位页面）
- 学习码卡片：渐变背景，引导用户查看个人二维码
- 热门活动列表：2×2 网格卡片，展示活动封面、标题、描述、参与人数
- 自习室推荐列表：封面图、名称、地址、营业状态、最低价格，分页加载
- 底部 TabBar：4 个 Tab（首页、预约、订单、我的）
- 后端提供轮播图、热门活动、自习室数据查询 API

**Non-Goals:**
- 预约功能（后续 change 实现）
- 自习室详情页
- 搜索、筛选、排序功能
- 后台管理端管理功能（数据暂通过数据库或脚本初始化）
- 微信授权登录（V2 考虑）
- 通知功能（铃铛图标仅展示，点击行为后续实现）

## Decisions

### 1. 首页数据获取策略：页面加载时并行请求

**选择**: 页面 `onShow` 时并行请求轮播图、热门活动、自习室列表三个接口。

**替代方案**: (A) 首屏只加载轮播图，其余懒加载；(B) 使用 Vuex 统一管理首页状态。

**理由**: 首页数据量小（轮播图 < 10，活动 < 10，自习室列表分页 10 条），并行请求减少首屏时间。Vuex 对当前简单首页过度设计，直接在页面组件内管理即可。

### 2. 自习室列表分页：后端分页 + 上拉加载更多

**选择**: 后端 `limit/offset` 分页，前端上拉触底加载更多。

**替代方案**: (A) 一次性返回全部自习室；(B) 前端虚拟列表。

**理由**: 自习室数量会增长，一次性返回不可扩展。前端虚拟列表对当前规模不必要。

### 3. 轮播图数据模型：独立 banners 表，支持标题/副标题/CTA 和跳转

**选择**: `banners` 表包含 `image_url`、`title`、`subtitle`、`cta_text`、`link_type`（none/page/room/url）、`link_value`、`sort_order`、`is_active` 字段。

**替代方案**: (A) 前端硬编码轮播图；(B) 使用 CMS 配置。

**理由**: 独立表便于运营管理，标题/副标题/CTA 按钮文案灵活可配。CMS 对 V1 来说过重。

### 4. 热门活动数据模型：独立 activities 表

**选择**: `activities` 表包含 `title`、`description`、`cover_image`、`participant_count`（展示用，可后续改为计算）、`is_active`、`sort_order` 字段。

**替代方案**: (A) 活动作为轮播图类型之一；(B) 前端硬编码活动卡片。

**理由**: 活动和轮播图是不同概念，独立表更清晰。后续活动管理、活动详情页等扩展更方便。

### 5. 自习室数据模型：study_rooms 表

**选择**: `study_rooms` 表包含 `name`、`description`、`cover_image`、`address`、`business_hours`、`status`（open/closed）、`min_price` 字段。

**理由**: 首页列表卡片展示所需的最小字段集，后续预约功能可扩展座位、区域等关联表。

### 6. TabBar 配置：原生 TabBar，4 个 Tab

**选择**: 使用 uni-app 原生 `pages.json` tabBar 配置，包含首页、预约、订单、我的四个 Tab。预约和订单页面 V1 创建占位页。

**替代方案**: (A) 自定义 TabBar 组件；(B) 3 个 Tab，订单放在"我的"页面内。

**理由**: 原型设计为 4 Tab，原生 TabBar 性能好且满足需求。预约和订单 Tab 创建简单占位页，后续逐步完善。

### 7. 图片托管方式：URL 字段手动配置

**选择**: `image_url` 字段存储完整 URL，图片上传到 OSS/CDN 后手动填入数据库。

**替代方案**: (A) 本地静态资源；(B) 集成文件上传 API。

**理由**: V1 阶段图片数量少且变动频率低，手动配置最简单。后续可补充管理后台的上传功能。

### 8. 快捷入口和学习码卡片：纯前端实现，无后端接口

**选择**: 快捷入口图标、名称、跳转路径前端硬编码；学习码卡片为纯 UI 组件，点击跳转到二维码页面。

**理由**: 快捷入口数量固定（4 个），内容不频繁变动，无需后端管理。学习码卡片是用户个人功能入口，V1 仅做跳转。

## Risks / Trade-offs

- **[自习室数据初始化]** → 首次部署需通过数据库迁移脚本插入示例数据，后续补管理后台
- **[并行请求失败]** → 各模块独立错误处理，单个接口失败不影响其他模块展示
- **[活动参与人数]** → V1 使用静态字段，后续需改为实时计算或定时更新
- **[快捷入口硬编码]** → 后续如需动态配置需增加后端接口

## Migration Plan

1. 新增数据库迁移：创建 `study_rooms`、`banners`、`activities` 表
2. 插入示例数据（至少 2 条轮播图、4 个活动、3 个自习室）
3. 部署后端新 API 路由
4. 更新小程序端首页和 TabBar 配置
5. **回滚**: 撤销迁移、移除新增路由文件、恢复原 `pages.json` TabBar 配置
